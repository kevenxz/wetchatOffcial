# 前端重设计方案：Brand Studio 工作台

## 1. 背景与目标

当前前端已经具备完整的公众号内容生产能力，包括任务创建、任务跟踪、历史资产、定时任务、风格配置、模型配置和账号配置，但整体界面仍然停留在通用后台模板层面：

- 全局骨架偏传统管理台，品牌感弱，难以体现“内容创作工作台”的定位。
- 页面主要依赖 Ant Design 默认 Card / Table / Form 组合，视觉层级单一。
- 高频页面的信息组织偏字段堆叠，无法突出“创作流程”和“结果资产”。
- 定时任务等复杂页面依赖长表单与模态框，操作负担较重。
- 部分中文文案存在编码异常，直接影响界面质感。

本次重设计的目标不是简单换皮，而是将前端统一升级为一个具备品牌识别度、创作流程感和现代化工作台体验的产品界面。

## 2. 产品定位

新前端定位为“品牌化创作工作台”，而不是传统中后台。

产品需要传达的核心感受：

- 用户不是在配置系统，而是在驱动一条内容生产链路。
- 任务创建、执行追踪、文章沉淀、自动发布应该形成连续体验。
- 系统配置存在，但不应抢占创作主线的视觉重心。

## 3. 设计方向

### 3.1 选定方向

采用 `Studio Canvas / Brand Studio` 方向作为统一视觉和交互方案。

方向关键词：

- 品牌化创作工作台
- 夜间工作室气质
- 内容生产链路可视化
- 状态驱动的现代控制感

### 3.2 记忆点

界面的记忆点需要集中在以下几个方面：

- 深墨黑与灰绿基底形成沉浸式工作区氛围
- 铜金与冷薄荷作为高辨识度强调色
- 带编辑感的大标题和强节奏排版
- 半透明面板、细颗粒纹理、柔和发光边缘
- 任务流和自动化规则具有明显“轨道 / 编排”感

### 3.3 明确避免

本次设计明确避免以下问题：

- 默认 Ant Design 管理台视觉
- 常见紫蓝渐变 SaaS 风格
- 单纯通过圆角、阴影、换色实现“现代化”
- 全站高密度表格化，没有内容展示感
- 只强调配置，不强调创作和结果

## 4. 信息架构调整

全站从“按页面堆功能”调整为“围绕创作主线组织入口”。

建议导航结构如下：

- 创作台
- 任务轨道
- 内容资产
- 自动化
- 系统舞台

对应现有功能映射：

- `任务创建` 作为创作台入口页
- `任务详情` 作为任务轨道页
- `历史任务` 与 `文章管理` 归入内容资产域
- `定时任务` 归入自动化域
- `风格配置`、`模型配置`、`账号配置` 归入系统舞台

这样调整后，用户对产品的理解会从“多个独立配置页”转为“围绕内容创作的工作区”。

## 5. 全局布局方案

### 5.1 全局骨架

全局采用三段式工作台骨架：

- 左侧固定品牌导航
- 顶部轻量上下文条
- 主内容沉浸式画布

### 5.2 左侧导航

左侧导航不再使用默认深色菜单视觉，而是承担品牌和分区导航职责：

- 顶部展示品牌标识和产品名称
- 中部展示主导航
- 底部展示系统设置类入口或辅助状态
- 折叠态仍保持图标辨识度和当前页面高亮

导航视觉应具备品牌气质，而不是纯功能菜单。

### 5.3 顶部上下文条

顶部区域不做传统后台 Header，而做上下文条：

- 展示当前页面标题和摘要
- 展示与当前页面相关的主操作
- 保留全局状态位，例如运行状态、连接状态、最近同步结果等

顶部条的目标是强化“当前正在做什么”，而不是提供重复导航。

### 5.4 主内容画布

主内容区统一采用工作台画布风格：

- 大页面顶部优先使用 Hero 区
- 主体内容使用 12 栏响应式网格
- 页面模块之间保留明显的节奏和分层
- 通过背景渐层、微纹理和模糊层构建空间感

## 6. 页面级重构

### 6.1 任务创建页：创作启动台

目标：让用户感觉自己在启动一次内容生产流程，而不是填写一张表单。

改造方案：

- 页面顶部使用 Hero 区展示页面主题、简短说明和主行动
- 左侧作为主输入区，承载关键词、目标角色、文章策略和风格偏好
- 右侧作为辅助区，展示热点建议、角色预设、默认风格、最近成功任务摘要
- 热点建议从普通 Tag 升级为更具引导感的可点击建议块
- 提交按钮升级为明显的主行动组件，文案强调“启动创作流程”

核心体验目标：

- 用户在首页即可理解系统会如何工作
- 表单输入具备创作气氛和内容引导，而不是后台填写感

### 6.2 任务详情页：任务轨道页

目标：持续回答用户三个问题：

- 现在执行到哪一步
- 为什么在执行这一步
- 已经产出了什么结果

改造方案：

- 顶部展示任务总览，包括关键词、状态、进度、当前节点、异常信息和主操作
- 主体区域拆为左右双栏
- 左栏展示流程轨道、步骤状态和中间产物
- 右栏展示文章预览、草稿链接、风格画像、策略解析和关键元数据
- 成功态和失败态不再只是默认 Result 组件，而是统一融入工作台状态面板

核心体验目标：

- 强化流程透明度
- 强化结果呈现
- 降低字段堆叠感

### 6.3 历史任务与文章管理：内容资产区

目标：把“历史记录”升级为“可回看、可筛选、可继续利用的内容资产”。

改造方案：

- 顶部统一为资产 Hero 区，支持搜索、筛选、视图切换
- 默认优先展示内容摘要卡片视图
- 同时提供高密度列表视图，用于快速管理
- 将任务状态、策略、角色、文章结果等信息整合为更有语义的资产卡片
- `历史任务` 与 `文章管理` 在视觉上保持统一，可通过标签页或视图切换组织

核心体验目标：

- 结果比任务 ID 更重要
- 保留管理效率，同时提升资产感和复用感

### 6.4 定时任务页：自动化编排台

目标：让用户感觉自己在编排自动化生产规则，而不是填写长表单。

改造方案：

- 列表页突出每条规则的运行状态、触发方式、目标账号、热点捕获摘要和下次执行时间
- 新建 / 编辑不再依赖大而长的模态框
- 改为独立工作区、全屏抽屉或页内编排面板
- 配置按四个区块组织：
  - 执行规则
  - 内容生成
  - 热点来源
  - 推送目标
- 热点捕获、平台列表和筛选条件使用更清晰的分区和摘要卡片呈现

核心体验目标：

- 自动化规则一眼可读
- 复杂表单具有明确结构
- 配置和结果之间建立因果关系

### 6.5 系统页：系统舞台

目标：统一风格但弱化存在感，不抢占创作主线。

改造方案：

- `风格配置`、`模型配置`、`账号配置` 统一进入 backstage 风格区域
- 页面仍使用相同设计系统，但减少戏剧化 Hero 和过强视觉强调
- 更强调可读性、配置稳定性和分区清晰度

核心体验目标：

- 与全站视觉一致
- 明确是后台支持能力，而不是主工作流

## 7. 视觉语言规范

### 7.1 色彩

建议色彩方向：

- 基底色：深墨黑、石墨灰、灰绿黑
- 强调色：铜金、冷薄荷
- 状态色：成功、警告、错误保留明确语义，但需要重新校准以适配暗色基底

原则：

- 不使用常见紫色渐变
- 不平均分配颜色权重
- 重点信息和关键状态通过强调色建立层次

### 7.2 字体

建议使用“标题有个性，正文可长读”的组合：

- 标题字体承担品牌和编辑感
- 正文字体承担表单、描述和表格阅读

原则：

- 避免 Inter、Arial、Roboto 等常见默认方案作为视觉主轴
- 标题字体可更具调性，但不能影响中文阅读
- 若引入 Web 字体，需要评估加载成本和可用性

### 7.3 背景与质感

背景不应是纯平色块，而应包含以下元素中的部分组合：

- 大面积低对比渐层
- 模糊光斑或空间光感
- 极轻颗粒纹理
- 半透明玻璃面板
- 柔和边缘高光

目标是营造工作室氛围，而不是制造噪音。

## 8. 组件系统设计

需要建立一层高于 Ant Design 的工作台组件系统。

核心组件建议如下：

- `WorkbenchShell`
  - 负责全局布局、侧栏、上下文条和主画布
- `HeroPanel`
  - 承载页面标题、摘要、主操作和关键信息
- `GlassCard`
  - 通用内容卡片
- `MetricCard`
  - 指标和摘要信息卡片
- `SignalCard`
  - 状态和运行信号卡片
- `SectionBlock`
  - 表单和信息区块容器
- `StatusRail`
  - 任务详情的流程轨道
- `AssetList`
  - 内容资产视图容器
- `AutomationRuleCard`
  - 自动化规则摘要卡片

这些组件的目标是让页面从“使用默认组件拼接”升级为“使用统一语言表达产品”。

## 9. 动效与交互规则

动效原则：

- 强调节奏，不追求炫技
- 页面进入时分层淡入和轻微上浮
- 卡片悬停时只做轻量边框和阴影变化
- 任务进度、运行状态、告警可使用细微脉冲和色带提示
- 避免大范围缩放、强反弹和廉价微交互

交互原则：

- 主操作始终明确且优先级清晰
- 复杂配置页面按区块分段
- 任何长内容都需要结构化分层，避免滚动疲劳
- 错误状态和异常状态需要在视觉上足够明确

## 10. 技术实现策略

### 10.1 技术边界

保留当前 `React + Ant Design` 技术栈。

实现策略为：

- Ant Design 继续作为基础交互控件来源
- 页面布局、视觉风格、卡片体系、状态组件和主题变量改为自定义工作台层
- 尽量避免为追求视觉而替换成熟的表单与表格能力

### 10.2 样式组织

建议样式层次如下：

- `variables.css`：色彩、间距、阴影、圆角、动画时长等设计变量
- 全局样式：背景、字体、滚动条、基础重置、主题入口
- 页面级样式：各页面 Hero、布局和专属模块
- 组件级样式：工作台组件本身

### 10.3 结构调整重点

优先调整的结构包括：

- `App.tsx` 的全局骨架和导航体系
- 当前全局主题与变量文件
- 各页面内联样式的大量抽离
- `TaskDetail` 页的结构化重排
- `ScheduleManage` 从大模态配置改为更清晰的主界面编排结构

## 11. 实施顺序

推荐实施顺序如下：

1. 重建全局设计系统
2. 重构全局骨架和导航
3. 重构任务创建页
4. 重构任务详情页
5. 重构历史任务与文章管理页
6. 重构定时任务页
7. 统一系统配置页视觉
8. 清理编码异常文案和零散样式

## 12. 风险与约束

已识别风险：

- 当前大量内联样式会阻碍视觉统一
- 复杂配置页如果仍保留模态框结构，体验提升有限
- 现有中文乱码问题需要同步清理
- 引入新字体和复杂背景时需要控制加载成本与可读性
- 暗色基底下需要重新验证表单、表格和状态色的对比度

约束：

- 不改变现有核心业务流程
- 不引入高风险的 UI 重写路线
- 保持现有页面功能可用与可维护

## 13. 验证标准

### 13.1 视觉验证

- 桌面端整体层级、空间感和品牌识别度显著提升
- 移动端主路径可用，重点保留任务创建、任务进度查看和结果查看
- 暗色主题下表单、文本、状态标签具备足够可读性

### 13.2 功能验证

- 任务创建流程可正常发起
- 任务详情可正常展示状态、进度和结果
- 历史任务和文章管理可正常浏览和筛选
- 定时任务可创建、编辑、启动、停止、立即执行
- 风格、模型、账号配置页可正常读取和提交

### 13.3 构建验证

- `frontend` 下执行 `npm run build`
- `frontend` 下执行 `npm run lint`

只有在视觉、功能和构建验证都通过后，才能将本次重设计视为完成。

## 14. 非目标

本次重设计不包含以下内容：

- 后端接口能力改造
- 工作流逻辑变更
- 新增业务模块
- 对现有内容生产策略的功能扩展

本次仅聚焦前端产品化表达、交互组织和视觉重构。
